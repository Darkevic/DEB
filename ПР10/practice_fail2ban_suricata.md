## Цель

Настроить fail2ban для защиты SSH от брутфорса, смоделировать реальную атаку через hydra на собственной виртуальной машине, увидеть как fail2ban банит атакующий IP, а затем познакомиться с Suricata: установить, написать собственное правило обнаружения брутфорса на сетевом уровне и увидеть алерт.

---

## Подготовка (10 мин)

Создайте папку практики:

```bash
cd ~/security-course
mkdir -p pr15/screens pr15/rules
touch pr15/report.md
```

Установите необходимые пакеты:

```bash
sudo apt-get update
sudo apt-get install -y fail2ban hydra suricata jq
```

Убедитесь что SSH-сервер у вас работает (он должен быть с первой практики):

```bash
sudo systemctl status ssh
```

Узнайте реальный IP-адрес своей виртуальной машины — он понадобится для всей практики, не используем 127.0.0.1:

```bash
ip addr show | grep "inet " | grep -v 127.0.0.1
```

Запишите этот адрес, далее в инструкции он обозначен как VM_IP.

Почему не localhost: по умолчанию fail2ban не банит 127.0.0.1 (прописано в ignoreip), поэтому атаковать нужно настоящий IP машины, а не loopback-адрес.

## Часть 1. Настраиваем fail2ban для SSH (20 мин)

### 1.1 Создаём тестового пользователя

Чтобы не рисковать основным аккаунтом, создадим отдельного пользователя специально для брутфорса:

```bash
sudo useradd -m -s /bin/bash bruteforce-target
echo "bruteforce-target:weakpass123" | sudo chpasswd
```

Это намеренно слабый пароль для учебного пользователя. Никогда не делайте так с реальными аккаунтами.

### 1.2 Создаём конфигурацию fail2ban

```bash
sudo nano /etc/fail2ban/jail.local
```

Содержимое файла:

```ini
[DEFAULT]
# не трогаем localhost при бане (но для атаки используем VM_IP, не 127.0.0.1)
ignoreip = 127.0.0.1/8 ::1

[sshd]
enabled = true
port = ssh
filter = sshd
backend = systemd
maxretry = 3
findtime = 600
bantime = 600
```

Перезапустите fail2ban и проверьте статус:

```bash
sudo systemctl restart fail2ban
sudo systemctl status fail2ban
sudo fail2ban-client status
sudo fail2ban-client status sshd
```

Сделайте скриншот вывода fail2ban-client status sshd. В отчёте: какие параметры показывает статус тюрьмы перед началом атаки?

### 1.3 Проверяем что фильтр настроен правильно

```bash
cat /etc/fail2ban/filter.d/sshd.conf | head -20
```

В отчёте: какое регулярное выражение использует фильтр sshd для поиска неудачных попыток (приведите хотя бы одну строку из файла)?

## Часть 2. Моделируем брутфорс через hydra (25 мин)

### 2.1 Готовим словарь паролей

Создадим словарь с заведомо неправильными паролями плюс правильный в конце — чтобы атака была реалистичной (много неудач, потом успех), но fail2ban должен забанить нас раньше чем мы дойдём до правильного:

```bash
cat > /tmp/passwords.txt << 'EOF'
123456
password
admin
qwerty
letmein
monkey
dragon
football
iloveyou
trustno1
weakpass123
EOF
```

Обратите внимание: правильный пароль weakpass123 стоит почти в конце списка, специально, чтобы fail2ban успел сработать раньше (maxretry=3) чем hydra доберётся до него.

### 2.2 Запускаем атаку

В одном терминале запустите наблюдение за статусом fail2ban в реальном времени:

```bash
watch -n 1 sudo fail2ban-client status sshd
```

В другом терминале запустите hydra против вашей VM (замените VM_IP на реальный адрес):

```bash
hydra -l bruteforce-target -P /tmp/passwords.txt -t 1 -V ssh://VM_IP
```

Флаг -t 1 означает один поток — это специально, чтобы атаки шли по очереди и легче было увидеть момент бана в реальном времени. Флаг -V показывает каждую попытку.

Наблюдайте за обоими терминалами. В какой-то момент hydra должна перестать получать ответы — это значит fail2ban забанил IP.

Сделайте скриншот терминала с hydra в момент когда атака "зависла" (соединения больше не устанавливаются).

Сделайте скриншот терминала с watch fail2ban-client status sshd показывающего забаненный IP.

### 2.3 Изучаем результат бана

Остановите hydra (Ctrl+C если ещё работает) и посмотрите детальный статус:

```bash
sudo fail2ban-client status sshd
```

В отчёте зафиксируйте:
- сколько всего неудачных попыток зафиксировано (Currently failed / Total failed)
- какой именно IP был забанен (Banned IP list)
- сколько было реальных банов

### 2.4 Смотрим логи fail2ban

```bash
sudo journalctl -u fail2ban | grep -i ban | tail -20
```

Найдите строку вида "Ban VM_IP". Сделайте скриншот. В отчёте: через сколько секунд после начала атаки произошёл бан? Совпадает ли это с настройками maxretry/findtime?

### 2.5 Проверяем что бан реально работает на уровне сети

```bash
sudo iptables -L -n | grep -A5 f2b
```

Вы увидите правило с забаненным IP в файрвол-таблице. В отчёте: через какой механизм fail2ban физически блокирует доступ?

Попробуйте проверить соединение на этот порт пока бан активен:

```bash
timeout 5 bash -c "echo > /dev/tcp/VM_IP/22" && echo "соединение прошло" || echo "соединение заблокировано"
```

### 2.6 Разбаниваем и пробуем снова

```bash
sudo fail2ban-client set sshd unbanip VM_IP
sudo fail2ban-client status sshd
```

Убедитесь что список забаненных IP пуст. Сделайте скриншот.

## Часть 3. Настраиваем более строгую защиту (10 мин)

Сейчас усилим защиту — уменьшим порог и увеличим время бана, как это бы сделал администратор после реальной атаки.

```bash
sudo nano /etc/fail2ban/jail.local
```

Измените параметры:

```ini
[sshd]
enabled = true
port = ssh
filter = sshd
backend = systemd
maxretry = 2
findtime = 300
bantime = 1800
bantime.increment = true
bantime.factor = 2
```

Параметр bantime.increment со множителем bantime.factor — прогрессивное увеличение времени бана при повторных нарушениях: первый бан 30 минут, при повторном нарушении после разбана — уже час, потом два часа и так далее. Это серьёзно отбивает желание долбиться повторно.

Примените изменения:

```bash
sudo systemctl restart fail2ban
sudo fail2ban-client status sshd
```

Повторите атаку из части 2.2 с тем же словарём (можно использовать ту же команду hydra) и убедитесь что бан срабатывает быстрее (после 2 попыток вместо 3).

Сделайте скриншот. В отчёте: на сколько секунд раньше сработал бан по сравнению с первой попыткой?

## Часть 4. Suricata — знакомство (20 мин)

### 4.1 Настраиваем интерфейс

Узнайте имя своего сетевого интерфейса (то же что в подготовке):

```bash
ip addr show
```

Откройте конфиг Suricata и проверьте HOME_NET — обычно по умолчанию там уже корректные приватные диапазоны, но интерфейс нужно указать явно:

```bash
sudo nano /etc/suricata/suricata.yaml
```

Найдите секцию af-packet и убедитесь что указан правильный интерфейс (замените eth0 на ваш, если другой):

```yaml
af-packet:
  - interface: eth0
```

### 4.2 Пишем своё правило обнаружения брутфорса

Создадим отдельный файл с собственным правилом, не трогая системные наборы:

```bash
sudo nano /etc/suricata/rules/local.rules
```

Содержимое:

```
alert tcp any any -> $HOME_NET 22 (msg:"Возможный SSH брутфорс - множество SYN на порт 22"; flow:to_server; flags:S; threshold:type both, track by_src, count 5, seconds 60; sid:1000001; rev:1;)

alert ssh any any -> $HOME_NET 22 (msg:"SSH соединение установлено - проверка протокола"; sid:1000002; rev:1;)
```

Подключите файл правил в основной конфиг:

```bash
sudo nano /etc/suricata/suricata.yaml
```

Найдите секцию rule-files: и добавьте туда - local.rules (рядом с уже существующими файлами наборов).

### 4.3 Проверяем конфигурацию и запускаем

```bash
sudo suricata -T -c /etc/suricata/suricata.yaml -v
```

Команда -T тестирует конфигурацию без реального запуска. Если есть ошибки в правиле — увидите их здесь, лучше исправить сразу.

Запустите Suricata:

```bash
sudo systemctl restart suricata
sudo systemctl status suricata
```

Сделайте скриншот статуса. В отчёте: запустилась ли служба без ошибок?

### 4.4 Генерируем трафик который должен сработать на правило

Запустите наблюдение за логом алертов в отдельном терминале:

```bash
sudo tail -f /var/log/suricata/fast.log
```

В другом терминале повторите hydra-атаку из части 2 (можно с другим словарём, можно с тем же):

```bash
hydra -l bruteforce-target -P /tmp/passwords.txt -t 1 ssh://VM_IP
```

Подождите немного и посмотрите в терминал с логом Suricata. Должна появиться строка с вашим правилом "Возможный SSH брутфорс".

Сделайте скриншот сработавшего алерта в fast.log.

Если правило не сработало сразу — проверьте что интерфейс в af-packet указан правильно и что Suricata реально видит трафик: sudo suricata --list-runmodes и sudo journalctl -u suricata | tail -30 помогут найти проблему.

### 4.5 Смотрим подробный JSON-лог

```bash
sudo tail -5 /var/log/suricata/eve.json | jq .
```

Найдите событие с типом alert и вашим sid (1000001). В отчёте: какие дополнительные поля есть в JSON-версии лога по сравнению с fast.log (например src_ip, dest_port, timestamp)?

## Шпаргалка

| Команда | Что делает |
|---------|-----------|
| sudo fail2ban-client status | Список активных тюрем |
| sudo fail2ban-client status sshd | Подробный статус конкретной тюрьмы |
| sudo fail2ban-client set sshd banip IP | Забанить IP вручную |
| sudo fail2ban-client set sshd unbanip IP | Разбанить конкретный IP |
| sudo fail2ban-client unban --all | Разбанить вообще все IP во всех тюрьмах |
| sudo fail2ban-client reload | Перечитать конфигурацию без перезапуска демона |
| hydra -l USER -P FILE ssh://IP | Перебор паролей по словарю против SSH |
| hydra -l USER -P FILE -t N -V ssh://IP | То же, с N потоками и подробным выводом |
| sudo suricata -T -c /etc/suricata/suricata.yaml | Проверить конфиг Suricata без запуска |
| sudo suricata-update | Скачать/обновить готовый набор правил |
| sudo tail -f /var/log/suricata/fast.log | Следить за алертами в реальном времени |
| jq . /var/log/suricata/eve.json | Красиво распарсить JSON-лог Suricata |
